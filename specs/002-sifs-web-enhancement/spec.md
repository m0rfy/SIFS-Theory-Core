# Feature Specification: Улучшение веб-версии SIFS Theory

**Feature Branch**: `002-sifs-web-enhancement`  
**Created**: 2025-01-27  
**Status**: Draft  
**Input**: User description: "Улучшение веб-версии SIFS Theory: создание полноценной веб-версии с роутингом, интеграцией всех документов, улучшенным дизайном в тематике SIFS, внедрением SSF-2025 Spatial Framework, концепцией технического документального музея, интеграцией всех интерактивных симуляций и использованием MCP для генерации изображений"

## User Scenarios & Testing _(mandatory)_

### User Story 1 - Навигация по веб-приложению (Priority: P1)

Пользователь открывает веб-приложение и хочет найти нужную информацию о теории SIFS. Он видит интуитивную навигацию внизу экрана, которая позволяет перейти к разным разделам: главная страница, теория, расчёты, предсказания, данные, симуляции. При наведении на категорию открывается подменю с конкретными документами или разделами.

**Why this priority**: Без навигации пользователь не сможет найти нужную информацию. Это базовая функциональность любого веб-приложения.

**Independent Test**: Можно протестировать, открыв приложение и проверив, что все основные разделы доступны через навигацию, подменю открываются корректно, и переходы между страницами работают.

**Acceptance Scenarios**:

1. **Given** пользователь открыл веб-приложение, **When** он видит навигацию внизу экрана, **Then** он может перейти к любому основному разделу (главная, теория, расчёты, предсказания, данные, симуляции)
2. **Given** пользователь навёл курсор на категорию в навигации, **When** он видит подменю с документами, **Then** он может выбрать конкретный документ для просмотра
3. **Given** пользователь находится на странице документа, **When** он использует навигацию, **Then** он может вернуться к индексу документов или перейти в другой раздел
4. **Given** пользователь использует мобильное устройство, **When** он открывает навигацию, **Then** она адаптируется под размер экрана и остаётся удобной для использования

---

### User Story 2 - Просмотр документации с визуальными эффектами (Priority: P1)

Пользователь открывает документ из раздела "Теория" или "Расчёты". Он видит красиво оформленный документ с математическими формулами, графиками, диаграммами и изображениями. Контент появляется плавно при прокрутке страницы. Формулы отображаются корректно, код подсвечен синтаксисом, изображения загружаются с эффектом появления.

**Why this priority**: Основная цель веб-приложения - предоставить доступ к документации. Без корректного отображения документов приложение не выполняет свою основную функцию.

**Independent Test**: Можно протестировать, открыв любой документ и проверив, что все элементы (текст, формулы, код, изображения, графики) отображаются корректно и красиво оформлены.

**Acceptance Scenarios**:

1. **Given** пользователь открыл документ, **When** он прокручивает страницу, **Then** контент появляется плавно с анимацией
2. **Given** в документе есть математическая формула, **When** пользователь просматривает документ, **Then** формула отображается в читаемом формате
3. **Given** в документе есть блок кода, **When** пользователь просматривает документ, **Then** код подсвечен синтаксисом и имеет кнопку копирования
4. **Given** в документе есть изображение, **When** пользователь просматривает документ, **Then** изображение загружается с эффектом появления и может быть увеличено при клике
5. **Given** в документе есть график или диаграмма, **When** пользователь просматривает документ, **Then** график отображается интерактивно и реагирует на действия пользователя

### User Story 3 - Интерактивные симуляции и расчёты (Priority: P1)

Пользователь переходит в раздел "Симуляции" и видит каталог доступных интерактивных симуляций: гравитационный коллапс, темпоральная калибровка, интерактивные расчёты, визуализации. Он выбирает симуляцию коллапса, настраивает параметры (масса, масштаб, скорость времени), запускает симуляцию и видит результаты в реальном времени. Он может сохранить параметры и результаты, сравнить с теоретическими предсказаниями.

**Why this priority**: Интерактивные симуляции - ключевая особенность веб-приложения, отличающая его от статичной документации. Это позволяет пользователям экспериментировать с теорией и понимать её на практике.

**Independent Test**: Можно протестировать, открыв любую симуляцию, настроив параметры, запустив её и проверив, что результаты отображаются корректно и все функции (сохранение, экспорт, сравнение) работают.

**Acceptance Scenarios**:

1. **Given** пользователь открыл раздел "Симуляции", **When** он просматривает каталог, **Then** он видит все доступные симуляции с описаниями и может выбрать нужную
2. **Given** пользователь открыл симуляцию коллапса, **When** он настраивает параметры (масса, масштаб, скорость времени), **Then** результаты обновляются в реальном времени
3. **Given** пользователь запустил симуляцию, **When** он видит результаты, **Then** он может сохранить параметры и результаты для последующего использования
4. **Given** пользователь получил результаты симуляции, **When** он просматривает их, **Then** он видит сравнение с теоретическими предсказаниями SIFS
5. **Given** пользователь использует интерактивные расчёты, **When** он вводит параметры, **Then** система вычисляет результаты и сравнивает их с экспериментальными данными

---

### User Story 4 - Визуальный опыт "Технического документального музея" (Priority: P2)

Пользователь открывает главную страницу и видит эпичную анимацию входа с визуальными эффектами. Он "путешествует" по "залам музея" (разделам), где каждый документ представлен как интерактивный экспонат. Информация раскрывается постепенно при прокрутке, создавая эффект storytelling. Пользователь видит примеры применения теории, которые показывают, как теория может изменить мир.

**Why this priority**: Визуальный опыт и эмоциональное вовлечение повышают интерес пользователей и помогают лучше понять теорию. Однако это улучшение UX, а не критическая функциональность.

**Independent Test**: Можно протестировать, открыв главную страницу и различные разделы, проверив, что анимации работают плавно, контент появляется постепенно, и визуальные эффекты создают нужное впечатление.

**Acceptance Scenarios**:

1. **Given** пользователь открыл главную страницу, **When** страница загружается, **Then** он видит эпичную анимацию входа с визуальными эффектами
2. **Given** пользователь прокручивает страницу документа, **When** он читает контент, **Then** информация раскрывается постепенно с анимациями (scroll-triggered animations)
3. **Given** пользователь просматривает раздел "Как теория изменит мир", **When** он изучает примеры, **Then** он видит визуализации применения теории в разных областях (энергетика, космос, медицина, связь)
4. **Given** пользователь перемещает курсор по странице, **When** он взаимодействует с элементами, **Then** он видит визуальные эффекты (частицы, свечение, анимации)
5. **Given** пользователь переходит между разделами, **When** он навигает по "залам музея", **Then** он видит плавные переходы с эффектом "путешествия во времени"

### User Story 5 - Адаптивный дизайн для всех устройств (Priority: P2)

Пользователь открывает веб-приложение на мобильном устройстве (смартфон или планшет). Все элементы интерфейса адаптируются под размер экрана: навигация упрощается, текст становится читаемым, графики и визуализации масштабируются, таблицы прокручиваются горизонтально или преобразуются в карточки. Пользователь может комфортно использовать все функции приложения на любом устройстве.

**Why this priority**: Большая часть пользователей использует мобильные устройства. Без адаптивного дизайна приложение будет неудобным для значительной части аудитории. Однако базовая функциональность должна работать и без идеальной адаптации.

**Independent Test**: Можно протестировать, открыв приложение на разных размерах экранов (от 320px до 1920px) и проверив, что все элементы отображаются корректно и остаются функциональными.

**Acceptance Scenarios**:

1. **Given** пользователь открыл приложение на мобильном устройстве (320px-768px), **When** он просматривает интерфейс, **Then** все элементы адаптированы под размер экрана и остаются функциональными
2. **Given** пользователь использует навигацию на мобильном устройстве, **When** он открывает меню, **Then** оно отображается в удобном для touch-интерфейса формате
3. **Given** пользователь просматривает документ на мобильном устройстве, **When** он читает контент, **Then** текст имеет оптимальный размер и читаемость
4. **Given** пользователь просматривает таблицу на мобильном устройстве, **When** он пытается её прочитать, **Then** таблица прокручивается горизонтально или преобразуется в карточки
5. **Given** пользователь просматривает график на мобильном устройстве, **When** он изучает данные, **Then** график масштабируется и остаётся читаемым

### User Story 6 - Динамическая визуальная среда, реагирующая на параметры теории (Priority: P2)

Пользователь использует интерактивную симуляцию, которая изменяет параметры теории SIFS (метрическая стабильность, временная дилатация, частота колебаний). Интерфейс приложения физически реагирует на эти изменения: блоки навигации вибрируют при низкой стабильности, фон "дышит" в такт с частотой колебаний, цвета индикаторов меняются в зависимости от временной дилатации. Пользователь видит прямую связь между параметрами теории и визуальным представлением.

**Why this priority**: Это уникальная особенность, которая делает приложение интерактивным и демонстрирует связь теории с визуальным опытом. Однако это улучшение UX, а не критическая функциональность.

**Independent Test**: Можно протестировать, запустив симуляцию, изменив параметры и проверив, что визуальные эффекты интерфейса реагируют на изменения параметров теории.

**Acceptance Scenarios**:

1. **Given** пользователь запустил симуляцию с параметрами теории, **When** он изменяет метрическую стабильность, **Then** блоки навигации вибрируют или стабилизируются в зависимости от значения
2. **Given** пользователь изменяет частоту колебаний в симуляции, **When** он наблюдает за интерфейсом, **Then** фон и тени "дышат" в такт с частотой
3. **Given** пользователь изменяет временную дилатацию, **When** он наблюдает за индикаторами, **Then** цвета индикаторов меняются от синего (стабильно) к красному (нестабильно)
4. **Given** пользователь использует несколько симуляций одновременно, **When** параметры изменяются, **Then** все визуальные эффекты интерфейса синхронизируются с параметрами

---

### User Story 7 - Поиск и фильтрация документов (Priority: P3)

Пользователь хочет найти конкретный документ или информацию. Он открывает индекс документов ("Карту музея") и использует поиск или фильтры по категориям. Он видит результаты поиска с preview и описаниями, может выбрать рекомендуемый маршрут (для новичков, для экспертов, для любопытных) и быстро найти нужную информацию.

**Why this priority**: При большом количестве документов поиск и фильтрация значительно улучшают пользовательский опыт. Однако базовая навигация через меню уже позволяет найти нужную информацию.

**Independent Test**: Можно протестировать, открыв индекс документов, выполнив поиск по ключевым словам, применив фильтры и проверив, что результаты корректны и полезны.

**Acceptance Scenarios**:

1. **Given** пользователь открыл индекс документов, **When** он вводит поисковый запрос, **Then** он видит релевантные документы с preview и описаниями
2. **Given** пользователь просматривает индекс документов, **When** он применяет фильтр по категории (теория, расчёты, предсказания), **Then** он видит только документы из выбранной категории
3. **Given** пользователь выбирает рекомендуемый маршрут (для новичков), **When** он следует маршруту, **Then** он видит последовательность документов, оптимальную для изучения теории
4. **Given** пользователь просматривает статистику документов, **When** он изучает индекс, **Then** он видит количество документов в каждой категории

---

### User Story 8 - Расширенные интерактивные расчёты для всех направлений теории (Priority: P1)

Пользователь открывает страницу интерактивных расчётов и видит калькуляторы для всех направлений теории SIFS: константы связи (G, α, α_s, G_F), тёмная энергия w(z), масса частиц, фрактальная структура, натяжение браны, квантовая запутанность, оптическая метрика, RS2 геометрия, зеркальные зоны, электрон как тор. Он вводит параметры, система вычисляет результаты, сравнивает их с экспериментальными данными (CODATA, DESI, EHT, LIGO) и показывает пошаговый вывод математических расчётов.

**Why this priority**: Расширенные расчёты позволяют пользователям экспериментировать со всеми аспектами теории и проверять её предсказания. Это критически важно для понимания и валидации теории.

**Independent Test**: Можно протестировать, открыв страницу расчётов, выбрав направление теории, введя параметры и проверив, что результаты корректны и сравниваются с экспериментальными данными.

**Acceptance Scenarios**:

1. **Given** пользователь открыл страницу интерактивных расчётов, **When** он просматривает доступные калькуляторы, **Then** он видит все направления теории SIFS (константы связи, тёмная энергия, масса частиц, фрактальная структура, натяжение браны, квантовая запутанность, оптическая метрика, RS2 геометрия, зеркальные зоны, электрон как тор)
2. **Given** пользователь выбрал калькулятор констант связи, **When** он вводит параметры масштаба S, **Then** система вычисляет G, α, α_s, G_F и сравнивает с экспериментальными данными
3. **Given** пользователь получил результаты расчёта, **When** он просматривает вывод, **Then** он видит пошаговое отображение математических выводов
4. **Given** пользователь выполнил расчёт, **When** он просматривает результаты, **Then** система показывает сравнение с экспериментальными данными (CODATA, DESI, EHT, LIGO) с указанием точности совпадения
5. **Given** пользователь выполнил несколько расчётов, **When** он просматривает историю, **Then** он может сравнить предыдущие расчёты с текущими

### User Story 9 - Страница "Как теория изменит мир" (Priority: P2)

Пользователь открывает страницу "Как теория изменит мир" и видит эпичную анимацию входа. Он "путешествует" по залам: будущее энергетики, путешествия во времени, квантовые вычисления, космические путешествия, медицина будущего, связь и коммуникации. В каждом зале он видит визуализации применения теории, сравнения "до" и "после", интерактивные примеры с симуляциями. Он понимает практическое значение теории и её потенциал для изменения мира.

**Why this priority**: Показ практического применения теории повышает интерес и понимание её значимости. Однако это улучшение UX, а не критическая функциональность.

**Independent Test**: Можно протестировать, открыв страницу "Как теория изменит мир", просмотрев все залы и проверив, что визуализации и примеры создают нужное впечатление.

**Acceptance Scenarios**:

1. **Given** пользователь открыл страницу "Как теория изменит мир", **When** страница загружается, **Then** он видит эпичную анимацию входа с визуализацией изменений
2. **Given** пользователь просматривает зал "Будущее энергетики", **When** он изучает примеры, **Then** он видит, как SIFS может революционизировать производство энергии с визуализациями и симуляциями
3. **Given** пользователь просматривает сравнение "до" и "после", **When** он изучает визуализации, **Then** он видит мир до и после применения теории с анимациями
4. **Given** пользователь просматривает интерактивный пример, **When** он взаимодействует с симуляцией, **Then** он видит практическое применение теории в реальном времени

### User Story 10 - SSF-2025 Spatial Framework с 4 уровнями реальности (Priority: P2)

Пользователь использует веб-приложение и видит, что интерфейс реализует SSF-2025 Spatial Framework с 4 уровнями реальности: Level 0 (Temporal Abyss - фон с частицами времени), Level 1 (Substrate - подложка страницы), Level 2 (Control Plane - навигация Orbital Dock), Level 3 (Pulse - индикаторы). Когда он запускает симуляцию, частицы времени в фоне ускоряются, блоки навигации вибрируют при низкой стабильности, фон "дышит" в такт с частотой колебаний. Он видит прямую связь между параметрами теории и визуальным представлением через Z-пространство.

**Why this priority**: SSF-2025 создаёт уникальный визуальный опыт, связывающий интерфейс с физическими параметрами теории. Однако базовая функциональность должна работать и без идеальной реализации SSF-2025.

**Independent Test**: Можно протестировать, запустив симуляцию, изменив параметры теории и проверив, что все 4 уровня реальности реагируют на изменения параметров.

**Acceptance Scenarios**:

1. **Given** пользователь открыл веб-приложение, **When** он просматривает интерфейс, **Then** он видит 4 уровня реальности SSF-2025 (Temporal Abyss, Substrate, Control Plane, Pulse)
2. **Given** пользователь запустил симуляцию с параметрами теории, **When** он изменяет временную дилатацию, **Then** частицы времени в фоне (Level 0) ускоряются, создавая ощущение "сжатия времени"
3. **Given** пользователь изменяет метрическую стабильность, **When** значение становится низким, **Then** блоки навигации (Level 2) вибрируют, а подложка страницы (Level 1) отбрасывает более размытую тень
4. **Given** пользователь изменяет частоту колебаний, **When** он наблюдает за интерфейсом, **Then** фон и тени "дышат" в такт с частотой, а индикаторы (Level 3) пульсируют с динамическим цветом

---

### Edge Cases

- Что происходит, когда пользователь открывает документ, который не существует или был удалён? Система должна показать понятное сообщение об ошибке и предложить вернуться к индексу документов.
- Как система обрабатывает очень длинные документы (более 10000 слов)? Документ должен загружаться постепенно, с ленивой загрузкой изображений и графиков, оглавлением для навигации.
- Что происходит, когда пользователь использует симуляцию с экстремальными параметрами (очень большие или очень маленькие значения)? Система должна валидировать параметры, показывать предупреждения и предотвращать некорректные расчёты.
- Как система обрабатывает медленное интернет-соединение? Изображения и графики должны загружаться с индикаторами загрузки, контент должен появляться постепенно.
- Что происходит, когда пользователь открывает приложение в браузере без поддержки современных функций (WebGPU, CSS переменные)? Система должна использовать fallback-решения (Canvas вместо WebGPU, статические стили вместо динамических переменных).
- Как система обрабатывает одновременное использование нескольких симуляций? Каждая симуляция должна работать независимо, но визуальные эффекты интерфейса должны синхронизироваться с активными параметрами.
- Что происходит, когда пользователь переключает язык интерфейса? Все тексты интерфейса должны переключаться, но содержимое документов остаётся на исходном языке (переводы документов - отдельная задача).

### Key Entities

- **Документ**: Представляет markdown файл из директории `docs/` с метаданными (название, описание, категория, путь, дата создания). Связан с другими документами через ссылки и категории.
- **Симуляция**: Интерактивный инструмент для экспериментирования с теорией SIFS. Имеет параметры (масса, масштаб, скорость времени), результаты расчётов, историю запусков. Связана с теоретическими предсказаниями для сравнения.
- **Маршрут навигации**: Определяет структуру переходов между страницами веб-приложения. Включает основные разделы (главная, теория, расчёты, предсказания, данные, симуляции) и подразделы (конкретные документы).
- **Визуальный эффект**: Динамический элемент интерфейса, реагирующий на параметры теории SIFS. Имеет тип (вибрация, свечение, анимация), параметры связи с теорией (метрическая стабильность, временная дилатация, частота колебаний).
- **Экспонат музея**: Представление документа или симуляции в концепции "технического документального музея". Имеет визуальное оформление, описание, связанные экспонаты, примеры применения теории.
- **SSF-2025 Level**: Один из 4 уровней реальности в SSF-2025 Spatial Framework. Level 0 (Temporal Abyss - фон с частицами времени), Level 1 (Substrate - подложка страницы), Level 2 (Control Plane - навигация), Level 3 (Pulse - индикаторы). Каждый уровень реагирует на параметры теории через CSS переменные.
- **Spatial компонент**: Компонент интерфейса, реализующий SSF-2025 Spatial Framework. Включает SpatialSlab, OrbitalDock, DeltaPulse, FrequencyKnob, ChronoOdometer, TemporalWave, SyncOrb, BentoCell, SpatialScrollbar, FractalDropdownMenu. Каждый компонент имеет связь с параметрами теории через JavaScript коннектор.
- **Расчёт направления теории**: Одно из направлений теории SIFS для интерактивных расчётов: константы связи, тёмная энергия, масса частиц, фрактальная структура, натяжение браны, квантовая запутанность, оптическая метрика, RS2 геометрия, зеркальные зоны, электрон как тор. Каждое направление имеет свои параметры, формулы и экспериментальные данные для сравнения.
- **Зал применения теории**: Один из залов на странице "Как теория изменит мир": будущее энергетики, путешествия во времени, квантовые вычисления, космические путешествия, медицина будущего, связь и коммуникации. Каждый зал содержит визуализации, примеры применения, сравнения "до" и "после".
- **Spatial Preset**: Готовый пресет SSF-2025 для применения в интерфейсе. Monolith (для основных страниц: центральный блок отделён от краёв, margin: 40px, border-radius: 40px, box-shadow: spatial-heavy, эффект уменьшения в перспективе при скролле), Orbital Dock (для навигации: островки с левитацией, каждая с частотой "левитации", вибрация при низкой стабильности), Data Capsule (для виджетов: градиентная обводка 0.5px, цвет меняется от синего к красному в зависимости от Time Dilation Delta).
- **JavaScript коннектор**: Утилита sifs-spatial-connector.ts для связи параметров теории SIFS с CSS переменными в реальном времени. Функция updateSpatialVariables(params) обновляет --sifs-metric-stability, --sifs-delta-color, --sifs-sigma-blur, --sifs-oscillation-speed, --sifs-time-dilation-delta. Интегрируется с существующими симуляциями автоматически. Использует requestAnimationFrame для плавных обновлений с батчингом. Предоставляет React хук useSpatialConnector().
- **Модульная структура файлов**: Архитектурное правило разбиения файлов более 300 строк на модули: основной файл (< 300 строк), файлы типов (.types.ts), файлы хуков (.hooks.ts), файлы утилит (.utils.ts), файлы конфигурации (.config.ts). Компоненты разделяются на логические части (Header, Body, Footer), страницы выносят секции в отдельные компоненты, утилиты группируются по функциональности.

## Assumptions

- Пользователи имеют современные браузеры с поддержкой JavaScript, CSS переменных и базовых веб-стандартов
- Пользователи имеют стабильное интернет-соединение для загрузки контента (изображения, графики, симуляции)
- Документы в директории `docs/` имеют корректный markdown формат и структуру
- Все изображения и ресурсы, необходимые для документов, доступны в директории `public/images/` или `docs/images/`
- Пользователи понимают базовые концепции теории SIFS или готовы изучать их через документацию
- Мобильные пользователи используют touch-интерфейс и ожидают адаптивного дизайна
- Пользователи могут использовать клавиатуру и мышь на десктопных устройствах
- Симуляции требуют вычислительных ресурсов браузера, и производительность зависит от устройства пользователя
- Переводы документов на другие языки будут добавлены позже отдельной задачей (сейчас только структура i18n для интерфейса)
- Браузеры 2025 года поддерживают WebGPU API для рендеринга частиц времени (с Canvas fallback для старых браузеров)
- Все используемые ресурсы будут бесплатными (MIT/ISC/CC0 лицензии) - shadcn/ui, Lucide React, Recharts, Motion, Tailwind CSS, готовые компоненты визуализации из проекта
- Технологический стек 2025 года обеспечивает высокую производительность: Tailwind CSS 4.1.12 для отрисовки тысяч изменений дельты в секунду, Motion 12.23.24 для сложных анимаций, Recharts 2.15.2 для визуализации темпоральных данных
